## 1. 全体像 — オープンコア(Open Core)

agent-start は **オープンコア** モデルを採用します。

- **コア(リポジトリの大部分)** = 無償・オープンソースの **MIT**
- **商用機能(`enterprise/` ディレクトリ)** = ソース閲覧可だが本番利用は有償の
  **Enterprise Edition(EE)ライセンス**

「個人がセルフホストで使う体験」は永久に無償(MIT)に保ち普及の燃料とし、
「チーム/規制対応組織でしか必要にならない機能」を EE に置いて収益化する、
という GitLab(`ee/`)・Sentry・PostHog と同型の構成です。

### 境界の判定ルール(最重要)

> **`enterprise/` 配下のファイルだけが EE。それ以外はすべて MIT。例外なし。**

機能フラグではなく**ディレクトリ(=構造)**で線を引いているため、コントリビューター
にも、GitHub のライセンス判定器にも曖昧さがありません。

| 対象 | ライセンス | できること |
| --- | --- | --- |
| `enterprise/` 以外すべて | [MIT](../LICENSE) | 利用・改変・セルフホスト・再配布(商用含む)が自由 |
| `enterprise/` 配下 | [EE License](../enterprise/LICENSE) | 閲覧・ビルド・評価・貢献は可。**本番利用は有償サブスク必須** |

## 2. MIT コア

ルートの [`LICENSE`](../LICENSE) は標準の MIT で、**`enterprise/` を除く
リポジトリ全体**に適用されます。著作権表記は `Copyright (c) 2026 imanect-labs
contributors`。

利用者は、著作権表記の保持(MIT の唯一の条件)さえ守れば、コアを自社製品に同梱
して販売することすら許されます。これがディストリビューション拡大のエンジンです。

GitHub のライセンス検出は、ルート LICENSE が純正 MIT のままなので「MIT」と表示
され続けます(EE はサブディレクトリの別ライセンスとして扱われる)。

## 3. Enterprise Edition ライセンス(`enterprise/LICENSE`)

**ソースアベイラブル(source-available)であって、オープンソースではありません。**
要点を条項単位で整理します。

| 条項 | 内容 |
| --- | --- |
| §1 定義 | Licensor=imanect-labs。「EE Software」= `enterprise/` 配下。「Subscription」=有償契約。「Production Use」=開発/テスト/評価**以外**の利用(社内ユーザ・顧客への機能提供を含む) |
| §2 非本番利用の許諾 | 開発・テスト・評価目的での閲覧/ビルド/実行、および貢献の提出を、無償・非独占・取消可能で許諾。**本番利用は含まない** |
| §3 本番利用の許諾 | 有効な Subscription がある期間・範囲でのみ本番利用可。契約終了で本番利用権も終了 |
| §4 制限 | サブスク無しの本番利用、EE コードを MIT コアや他プロジェクトへ移して回避する行為、無断の再配布/転売/第三者ホスティング、ライセンス表記の除去、商標の無断使用を禁止 |
| §5 貢献 | EE への貢献を提出すると、Licensor に対し永続・全世界・無償・取消不能の利用/改変/商用化ライセンスを付与。重要な貢献には別途 CLA を求める場合あり |
| §6 終了 | 違反で自動終了。終了後はあらゆる利用を停止し複製を破棄。§4〜§8 は存続 |
| §7 無保証 / §8 責任制限 | MIT と同水準の AS IS・責任制限 |

商用問い合わせ先は **`goto@imanect.co.jp`**(PR #100 で確定)。

### EE に置く予定の機能(`enterprise/README.md`)

現時点でコードは未実装で、ディレクトリは**境界の確立のみ**が目的です。予定スコープ:

- **認証** — SSO / SAML / OIDC、セッション管理
- **RBAC** — ロール、プロジェクト/エージェント単位の権限
- **監査ログ** — 誰がどのエージェントに何のプロンプトでどのリポを操作したか
- **ポリシー強制** — `--dangerously-skip-permissions` の許可制御、シークレット/ネットワーク境界
- **マルチホスト・オーケストレーション** — 複数ホスト/開発者を 1 コンソールで
- **コスト/利用の可観測性** — エージェント実行コスト集計とダッシュボード

> 戦略メモ:現在 README にある「組み込み認証なし」という弱点が、逆に**最初の有償
> SKU(認証・監査ログ)への最短の入口**になります。

## 4. 貢献のライセンス:DCO 採用(PR レビュー対応)

CodeRabbit のレビュー指摘(「CLA に言及があるが実体が無い」)を受け、コアには
**DCO(Developer Certificate of Origin)** を採用しました。

- コミットに `Signed-off-by:` を付ける軽量方式(`git commit -s`)。Linux / GitLab
  コアと同じ運用で、署名済み CLA の摩擦を避けつつ「貢献者が提出権を持つ」ことを
  証明させる
- 全文は [`DCO`](../DCO)(Linux Foundation の DCO 1.1 をそのまま収録)
- `enterprise/` への重要な貢献については、EE License §5 の貢献条項に加え、必要に
  応じて **CLA** を追加で求める(商用提供の権利を確実に確保するため)

`CONTRIBUTING.md` にサインオフ手順を、`LICENSING.md` の Contributing 節に方針を
反映済み。

## 5. 不可逆性についての保証

オープンコアで最も信頼を損なうのは「後から MIT を巻き戻す」行為です。本体制では
明文で次を保証しています(`LICENSING.md`):

- 境界は**構造的**(=`enterprise/` ディレクトリ)であり、隠れた機能フラグではない
- **リリース済みの MIT バージョンは永久に MIT**。既存の MIT コードをパイウォールの
  裏へ移すことはしない
- 新しい商用機能は**最初から `enterprise/`** に入れる

## 8. 戦略上の含意(まとめ)

- **無償の個人ユーザからは課金しない**。課金源は「チーム」と「コンプライアンス」。
  EE スコープ(SSO/RBAC/監査/マルチホスト/ポリシー)はすべてそこに紐づく
- セルフホスト/データ主権という、クラウド純正エージェント(Claude Code on the
  web / Codex cloud)が構造的に提供できない領域が堀
- ライセンス境界を**コードが大きくなる前の今**確定できたのが、この体制の最大の価値
